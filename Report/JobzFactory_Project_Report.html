General Introduction

Summary. This report presents the work carried out on JobzFactory, a job board platform developed as a multi-project ASP.NET MVC solution. The introduction outlines the motivation behind the project, the main actors involved, and the structure of the document.

Online recruitment has become a standard channel for both companies and job seekers. Small and medium businesses often need a straightforward tool to publish vacancies, receive applications, and track candidates without relying on expensive third-party platforms. JobzFactory was built to address this need: it connects recruiters, candidates, and administrators through several linked web applications that share a single SQL Server database.

The solution is not a single website but a Visual Studio solution containing five .NET projects plus a database project. The public-facing site allows anyone to browse published offers and apply with a CV. The recruiter portal gives authenticated users the ability to create offers, publish them, and review applicants. A separate candidate registration module handles profile creation. An administration module provides full CRUD operations on recruiters and offers for back-office use.

During this project, most of my effort went into understanding the existing codebase, setting up the local database, tracing data flows between controllers and entities, and documenting how the different portals interact. Some features—such as email verification for candidates—exist in the code but are partially disabled or commented out, which is discussed later in the report.

This document is organised into five chapters. Chapter 1 defines the context and objectives. Chapter 2 describes the architecture. Chapter 3 covers the database. Chapter 4 walks through each application module. Chapter 5 focuses on the implementation work, challenges, and points for improvement. A general conclusion closes the report, followed by the bibliography and annexes.

Conclusion. JobzFactory is a practical recruitment platform whose value lies in its modular structure and shared data layer. The following chapters explain how the system was designed and what was achieved during the project period.

Chapter 1 — Project Context and Objectives

Chapter summary. This chapter explains why JobzFactory was developed, what problems it is meant to solve, and what was included or excluded from the project scope.

1.1 Background and Problem Statement

Traditional recruitment often involves scattered communication: offers posted on social media, CVs received by email, and no central history of changes to a vacancy. For a development project in a web technologies course (or internship setting), the goal was to build something closer to real industry practice—a system with distinct user roles, persistent storage, and a maintainable structure.

JobzFactory targets three types of users. Recruiters need to manage job postings and see who applied. Candidates need a simple way to submit an application with an attached CV. Administrators need oversight of all recruiters and offers. The public website ties everything together by displaying only published vacancies and routing applications into the database.

The existing codebase already contained a working foundation when the project was taken over. My role was to analyse it thoroughly, ensure the database could be recreated locally, and clarify how each module contributed to the overall platform.

1.2 Project Goals

The main objectives assigned to this project can be summarised as follows:

  1. Understand and document the full solution architecture (web projects, data layer, database).
  2. Ensure the SQL Server schema matches the Entity Framework model and can be deployed via scripts or DACPAC.
  3. Describe the functional behaviour of each portal (public site, recruiter, candidate, admin).
  4. Identify security weaknesses and incomplete features for future correction.
  5. Prepare the solution for build and deployment using Azure DevOps pipeline configuration already present in the repository.

Secondary goals included verifying that sample data and test accounts work for demonstration purposes, and mapping uploaded CV files to their corresponding application records in the database.

1.3 Scope and Limitations

The project scope covers the backend logic, database design, and MVC views contained in the repository. It does not include mobile applications, payment processing, or advanced matching algorithms between candidates and offers.

Several limitations should be noted honestly. Password storage uses plain text comparison in the login controller, which is not acceptable for production. Email sending for account activation is implemented in helper classes but largely commented out in the registration flow. The administration portal does not implement visible authentication in the controller layer, meaning it should not be exposed publicly without additional security work.

These limitations do not invalidate the project as a learning exercise, but they define clear boundaries for what the current version can safely be used for—mainly local development, demonstration, and further hardening.

Chapter conclusion. JobzFactory addresses a real recruitment workflow with a multi-portal architecture. The project goals focused on understanding, documenting, and validating the system rather than rewriting it from scratch. Scope was deliberately kept aligned with what the existing solution already supports.

2.3 Data Flow Between Portals

The following table describes a typical end-to-end scenario: a candidate applies to a job that was published by a recruiter. This helps clarify why all portals must share the same database rather than maintaining separate stores.

Chapter 3 — Database Design

Chapter summary. This chapter details the relational schema, the main entities, and how the database is created and populated for local development.

3.1 Entity Model

The Entity Framework model is defined in JF.DAL/Context/Model.edmx and was reverse-engineered from the SQL Server schema (or kept in sync with it). Fourteen entity sets map to tables in the dbo schema. The entity-relationship structure is summarised below.

Pays ──► Ville ──► Profil / Offre
SecteurActivite ──► Profil / Offre
Recruteur ──► Recruteur_Contact
Recruteur ──► Offre ──► Offre_Historique (via ActionOffre)
Offre ──► Offre_Postuler ◄── Profil
Profil ──► Profil_CV, Profil_Link
Codification ──► Offre (offer numbering)

Figure 2 — Simplified entity relationships in the jobzFactory database (source: Database/README.md and Model.edmx, 2026)

The most heavily used tables during normal operation are Offre, Recruteur, Recruteur_Contact, Profil, and Offre_Postuler. Reference data lives in Pays, Ville, and SecteurActivite. The Offre_Historique table records actions performed on offers (useful for the recruiter history view).

3.2 Deployment and Seed Data

Two approaches exist to create the database, which is helpful when working on different machines or preparing a demo environment.

SQL scripts. Running jobzFactory_CreateDatabase.sql, then jobzFactory_SeedData.sql, then jobzFactory_SampleData.sql in order will build the schema and insert reference and demo data. This method is easy to follow in SSMS.

DACPAC (recommended in project docs). The SSDT project under Database/jobzFactory.Database compiles to a DACPAC file. The PowerShell scripts Build-Dacpac.ps1 and Publish-LocalDatabase.ps1 automate build and publish to a local SQL instance. Post-deployment script Script.PostDeployment.sql loads seed and sample data automatically.

Sample accounts documented in the database README include recruiter logins recruteur.demo and tech.hr (password 123), and candidate email candidat.demo@test.local. These accounts were used during testing to verify login and listing behaviour without creating new data each time.

All web projects share the same connection string pattern: Entity Framework metadata paths pointing to embedded CSDL/SSDL/MSL resources, with a SQL provider connection to data source=.;initial catalog=jobzFactory;integrated security=True.

Chapter conclusion. The database is relational, normalised around recruiters, offers, and applications, with supporting lookup and audit tables. Deployment is well supported through both manual scripts and SSDT automation, which made local setup repeatable during the project.

Chapter 4 — Application Modules

Chapter summary. This chapter describes the four web applications: what each one does, which controllers and views matter most, and how they fit into the overall platform.

4.1 Public Website (JobzFactory)

The main site is the public entry point. Its homepage (DefaultController.Index) loads published offers with pagination (six per page), including related city and recruiter data. Offers are ordered by ID descending as a stand-in for creation date ordering.

JobController handles the core candidate journey. The Detail action resolves an offer by URL slug. The Apply action shows a form; on POST it creates a new Profil record, an Offre_Postuler record, and saves the uploaded CV to a folder named after the application ID. A confirmation view (SendApply) is shown only when redirected with the correct TempData flag, which avoids accidental resubmission on refresh.

Additional controllers (ProfilController, RecruteurController) expose public profile and recruiter detail pages. Static assets include a full Bootstrap-based theme, carousel scripts, and a large collection of uploaded CVs under Ressource/CV/OffrePostule/ from real test submissions.

4.2 Recruiter Portal (Recruteur)

Recruiters authenticate through LoginController using Forms Authentication. On successful login, user details are serialised into the auth ticket and a static session helper (VarSessionRecruteur) holds the current contact for the request pipeline.

The AnnonceController is the heart of this portal. It lists offers belonging to the logged-in recruiter, supports create/edit/delete, manages publication status, shows a kanban-style board view, lists candidates per offer, and displays paginated action history. Access is restricted by [CustomAuthorize(Roles = "RecruteurAdmin")].

New recruiters can self-register via LoginController.SignUp, which creates both a Recruteur entity and a linked Recruteur_Contact with a randomly generated password. The UI uses a dashboard theme (CSS/JS under assets/) distinct from the public site’s look and feel.

4.3 Candidate Portal (Profil)

The Profil project is smaller in scope. SignUpController.Register accepts candidate details, checks for duplicate email, inserts a Profil row, and creates a Profil_Link verification token valid for ten minutes. The email sending and account activation actions exist but are commented out in the controller, so registration currently ends at the EndSignUp view without actual email confirmation.

Default password 123 is hardcoded at registration—the same pattern appears in the public apply flow. This was likely chosen for prototyping speed but should be changed before any real deployment.

4.4 Administration Module

Administration provides scaffold-style CRUD for Offre and Recruteur through OffreController and RecruteurController. Views follow standard MVC patterns: Index lists, Create/Edit forms with dropdowns for foreign keys, Details and Delete confirmation pages.

Unlike the recruiter portal, no authorisation attribute appears on these controllers in the current codebase. For a production system, this module would need at least role-based access control and ideally integration with the same authentication mechanism used elsewhere—or a separate admin identity store.

Chapter 5 — Implementation Work and Challenges

Chapter summary. This chapter focuses on what was actually done during the project, the problems encountered while working with the codebase, and recommendations for security and future development.

5.1 Work Completed During the Project

The following items represent the main deliverables and tasks completed:

  • Full inventory and analysis of the solution: six projects, roughly 650 source files excluding build output.
  • Review of all C# controllers and their interaction with jobzFactoryEntities.
  • Documentation of database tables, relationships, and deployment options (scripts vs DACPAC).
  • Verification of connection strings across Web.config and App.Config files for local SQL Server.
  • Mapping of the job application flow from public apply form to file storage and recruiter candidate view.
  • Review of Azure Pipelines YAML for automated build configuration.
  • Preparation of this report structured according to academic formatting requirements.

Where the repository already contained working features, the priority was understanding and documenting rather than rewriting. That approach saved time and respected the existing architecture, which is a reasonable strategy when taking over a inherited project.

5.2 Difficulties Encountered

Several issues came up during analysis and local setup:

Database-first and EDMX. Entity Framework Database First with Model.edmx is less common in newer projects than Code First. Any schema change requires updating SQL, refreshing the model, and rebuilding JF.DAL—a process that is easy to get wrong if steps are skipped.

Hardcoded values. The offer numbering logic in AnnonceController.Create references year 2022 in one LINQ filter regardless of the current year, which can produce incorrect codification counts over time. Password defaults and plaintext storage were surprising to find and required explicit documentation.

Disabled email features. Tracing why registration did not send verification emails led to commented-out code in SignUpController and unused email helper classes. This is not a bug in the strict sense, but it can confuse anyone expecting the feature to work out of the box.

Multiple web roots. Hosting four separate MVC apps locally means configuring four IIS sites or using multiple startup projects in Visual Studio, each with its own port. This is standard for the architecture but adds setup overhead compared to a single monolithic app.

Chapter conclusion. The project phase delivered solid understanding and documentation of JobzFactory. The main technical debt lies in security and incomplete features rather than in the overall architecture, which remains coherent for continued development.

General Conclusion

JobzFactory demonstrates how a recruitment platform can be split into specialised web applications while sharing a single data layer and database. The public site, recruiter portal, candidate registration module, and administration back-office each address a clear part of the hiring workflow.

Through this project, I gained practical experience reading a multi-project ASP.NET MVC solution, tracing Entity Framework queries from controllers to tables, and deploying a SQL Server schema using both scripts and SSDT. The platform is functional for demonstration: recruiters can manage offers, candidates can apply with CV uploads, and administrators can inspect records through CRUD screens.

At the same time, the analysis highlighted areas that must be addressed before any production release—password handling, admin access control, and secrets management being the most urgent. Email verification and offer numbering also need attention to match the intended design.

Overall, JobzFactory is a workable foundation. The architecture is understandable, the database is well structured, and the separation of portals maps cleanly to real user roles. Future work should focus on security hardening, completing partially implemented features, and optionally modernising the stack (for example, migrating toward ASP.NET Core and Code First over time).

This report aimed to document the system honestly, with emphasis on what was built, how it fits together, and what remains to be done—so that anyone reading it can continue the project without starting from zero.
